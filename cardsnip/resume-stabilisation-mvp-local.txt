Resume - Stabilisation MVP local CardSnip

Objectif de cette passe
-----------------------

Stabiliser le flux local existant sans ajouter de nouvelle feature, sans nouvelle page, sans vraie boutique, sans Supabase et sans refonte UI.

Flux vise :

fake-shop -> scraper Python -> SQLite -> FastAPI -> dashboard -> alertes


Ce qui a ete fait
-----------------

1. Verification du flux "Creer un suivi" depuis Catalogue

Le flux a ete verifie via les memes endpoints utilises par le formulaire Catalogue :

- POST /products
- GET /shops
- POST /tracked-products

Resultat :

- un produit catalogue peut etre cree dans SQLite ;
- une boutique Fake Shop peut etre selectionnee ;
- un suivi peut etre cree dans tracked_products ;
- aucune manipulation SQL manuelle n'est necessaire pour ce flux applicatif.


2. Page Produits branchee sur les observations reelles

Fichier modifie :

web/src/components/pages/products-page.tsx

Avant :

- la page Produits lisait les tracked_products depuis l'API ;
- mais elle n'affichait pas encore les derniers prix/stocks reels ;
- elle pouvait retomber sur les deals mockes, ce qui creait une confusion entre donnees reelles et donnees demo.

Apres :

- la page Produits lit maintenant aussi GET /observations/latest ;
- chaque produit suivi SQLite est associe a sa derniere observation si elle existe ;
- le prix actuel affiche le dernier prix reel observe ;
- le stock detecte s'affiche quand il existe ;
- si l'API locale est disponible, la page affiche les suivis SQLite au lieu de melanger avec les deals mockes.

Objectif :

Rendre la page Produits plus coherente avec le vrai flux MVP local.


3. Reduction du melange mocks / reel

Sur la page Produits :

- API disponible = affichage des suivis SQLite ;
- API indisponible = fallback local/mock.

Cela evite de faire croire que des deals mockes sont des donnees reelles quand l'API locale fonctionne.


4. Deduplication simple des alertes

Fichier modifie :

scraper/services/tracked_check_service.py

Avant :

- chaque lancement du scraper pouvait recreer une alerte price_below_target si le produit restait sous le prix cible.

Apres :

- si l'observation precedente etait deja en stock et sous le target_price, le scraper ne recree pas une alerte identique au check suivant ;
- l'alerte est generee uniquement quand le produit passe sous la cible, ou lors du premier check pertinent.

Objectif :

Eviter les doublons d'alertes inutiles pendant les tests locaux.


5. Script de test end-to-end local

Fichier ajoute :

scripts/test-end-to-end-local.ps1

Role :

Automatiser le test du flux local complet.

Le script :

- initialise SQLite avec python scripts/init_db.py ;
- verifie que la fake shop repond ;
- verifie que FastAPI repond ;
- teste le scenario price=39.99&stock=in ;
- lance POST /scraper/run ;
- verifie que l'observation vaut 39.99 et in_stock ;
- teste le scenario price=99.99&stock=out ;
- lance POST /scraper/run ;
- verifie que l'observation vaut 99.99 et out_of_stock ;
- restaure l'URL demo initiale price=39.99&stock=in.

Commande :

cd "C:\Users\Nagara\Documents\New project\cardsnip"
.\scripts\test-end-to-end-local.ps1

Precondition :

La fake shop et FastAPI doivent deja etre lancees.

Limites restantes
-----------------

- Le frontend reste partiellement mocke sur certaines pages.
- La page Produits est amelioree mais pas encore une UX finale.
- Pas de tests automatises unitaires.
- Pas de vraie boutique.
- Pas de Supabase branche.
- Pas de vraie gestion d'images produit.
- Pas de deduplication avancee des alertes.
- Pas de systeme de planification automatique du scraper.


Prochaine etape recommandee
---------------------------

Avant toute nouvelle feature :

1. Continuer a nettoyer les derniers melanges mock/reel.
2. Ajouter quelques tests backend simples autour des repositories et du scraper.
3. Verifier le flux complet depuis l'interface Catalogue avec navigation utilisateur.
4. Clarifier les donnees seed/demo.
5. Preparer ensuite seulement un premier vrai scraper autorise.

Ne pas faire tout de suite :

- Supabase ;
- auth ;
- Stripe ;
- vraie boutique ;
- Docker ;
- Celery ;
- workers ;
- nouvelle page frontend.
